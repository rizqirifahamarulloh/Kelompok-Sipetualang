# AI Agentic Rules — Backend (Laravel 13)

> Rules ini WAJIB dipatuhi oleh semua AI agent (Antigravity, Cursor, Copilot, atau IDE-based AI) saat bekerja di folder `backend/`.

---

## 0. ⚠️ MANDATORY: Artisan Commands First

> **SELALU gunakan `php artisan make:*` commands untuk generate file Laravel.**
> **JANGAN PERNAH manually create file yang bisa di-generate oleh artisan.**

### Wajib pakai artisan:
| Kebutuhan | Command |
|-----------|--------|
| Migration | `php artisan make:migration create_xxx_table --create=xxx` |
| Model | `php artisan make:model ModelName` |
| Model + all | `php artisan make:model ModelName -a` (model+migration+seeder+factory+policy+controller+formrequest) |
| Controller | `php artisan make:controller Api/Admin/XxxController --api` |
| Form Request | `php artisan make:request StoreXxxRequest` |
| API Resource | `php artisan make:resource XxxResource` |
| Seeder | `php artisan make:seeder XxxSeeder` |
| Middleware | `php artisan make:middleware XxxMiddleware` |
| Policy | `php artisan make:policy XxxPolicy --model=Xxx` |
| Test | `php artisan make:test Xxx/XxxTest` |
| Enum | `php artisan make:enum XxxEnum` (jika tersedia) |

### Kapan boleh manual write:
- Hanya jika artisan command **tidak tersedia** untuk file tersebut
- Contoh: Traits, Interfaces, Services, Enums (jika make:enum tidak ada)
- Setelah artisan generate file → **edit isi file** sesuai kebutuhan

### Contoh workflow:
```bash
# ✅ BENAR
php artisan make:migration add_avatar_to_users_table --table=users
php artisan make:controller Api/Admin/UserController --api
php artisan make:resource UserResource
php artisan make:request StoreUserRequest

# ❌ SALAH — jangan manually create file
touch database/migrations/2026_05_02_000001_xxx.php  # DILARANG
```

## 1. Commit Convention — Conventional Commits

Semua commit WAJIB mengikuti format:

```
<type>(<scope>): <description>

[optional body]
[optional footer]
```

### Types:
| Type | Usage |
|------|-------|
| `feat` | Fitur baru |
| `fix` | Bug fix |
| `refactor` | Refactoring tanpa mengubah behavior |
| `docs` | Dokumentasi |
| `test` | Menambah atau memperbaiki test |
| `chore` | Maintenance (deps, config, etc.) |
| `style` | Formatting, semicolons, etc. (bukan CSS) |
| `perf` | Performance improvement |
| `ci` | CI/CD changes |

### Scopes:
`auth`, `user`, `gear`, `category`, `destination`, `transaction`, `payment`, `deposit`, `migration`, `seeder`, `config`, `middleware`, `route`

### Contoh:
```
feat(auth): implement JWT login and register endpoints
fix(transaction): fix overdue status scope query
refactor(gear): extract availability check to scope
```

## 2. SOLID Principles

### Single Responsibility (SRP)
- 1 Controller = 1 Resource
- Business logic di **Service class**, bukan di Controller
- Validation di **Form Request**, bukan di Controller
- Data transformation di **API Resource**, bukan di Controller

### Open/Closed (OCP)
- Gunakan **interfaces** untuk service yang mungkin berubah implementasi
- Gunakan **config/enum** untuk values yang mungkin berubah

### Liskov Substitution (LSP)
- Jangan override method parent yang mengubah contract

### Interface Segregation (ISP)
- Interfaces harus spesifik, bukan general-purpose

### Dependency Inversion (DIP)
- Controller depend on Service interface, bukan concrete class
- Bind di `AppServiceProvider`

## 3. Laravel 13 Best Practices

### PHP Attributes (Laravel 13 Feature)
```php
// ✅ Gunakan PHP Attributes untuk Fillable & Hidden
#[Fillable(['name', 'email', 'password'])]
#[Hidden(['password', 'remember_token'])]
class User extends Authenticatable
```

### Strict Types
```php
// ✅ WAJIB di setiap file PHP
declare(strict_types=1);
```

### Naming Convention
| Element | Convention | Example |
|---------|-----------|---------|
| Model | Singular PascalCase | `Gear`, `TransactionDetail` |
| Table | Plural snake_case | `gears`, `transaction_details` |
| Controller | Singular + Controller | `GearController` |
| Migration | Artisan-generated timestamp | `2026_05_01_225918_create_gears_table` |
| Seeder | Singular + Seeder | `GearSeeder` |
| Request | Action + Resource + Request | `StoreGearRequest` |
| Resource | Singular + Resource | `GearResource` |
| Policy | Singular + Policy | `GearPolicy` |

### Route Convention
```php
// ✅ Gunakan apiResource untuk CRUD
Route::apiResource('gears', GearController::class);

// ✅ Group by role middleware
Route::middleware('role:super_admin')->group(function () { ... });
```

### Query Best Practices
```php
// ✅ Gunakan Eloquent scopes
Gear::available()->byOwner($userId)->get();

// ✅ Eager load relationships
Transaction::with(['customer', 'details.gear'])->get();

// ❌ JANGAN N+1 query
foreach ($transactions as $trx) {
    $trx->customer->name; // N+1!
}
```

### Error Handling
```php
// ✅ Gunakan ApiResponse trait
return $this->error('Resource not found', 404);
return $this->success($data, 'Created successfully', 201);

// ✅ i18n ready error messages
return $this->error(__('auth.failed'), 401);
```

## 5. API Response Format

Semua API response WAJIB menggunakan format konsisten:

```json
{
  "success": true,
  "message": "Success message",
  "data": { ... }
}
```

Error format:
```json
{
  "success": false,
  "message": "Error message",
  "errors": { ... }
}
```

Paginated format:
```json
{
  "success": true,
  "message": "Success",
  "data": [ ... ],
  "meta": {
    "current_page": 1,
    "last_page": 5,
    "per_page": 15,
    "total": 75
  }
}
```

## 6. Testing Requirements

- Setiap fitur baru HARUS punya minimal 1 feature test
- Auth endpoint harus di-test untuk semua role
- Gunakan `RefreshDatabase` trait
- Test file naming: `Feature/Auth/LoginTest.php`

## 7. Security Rules

- JANGAN expose sensitive data di response (password, tokens)
- SELALU validate input via Form Request
- SELALU gunakan `auth:api` middleware untuk protected routes
- SELALU gunakan Spatie role/permission middleware
- JANGAN hardcode credentials — gunakan `.env`
